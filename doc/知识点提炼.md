## 项目描述
```text
基于 Spring Boot + Spring AI + DashScope OpenAI 兼容接口，围绕智慧食堂 / 明厨亮灶监管场景，构建 AI 应用工程化 Demo。系统支持监管问题问答、RAG 知识库检索、AI 调用日志、知识未命中复盘、知识库管理、Embedding 向量持久化、知识变更审计、Function Calling 工具调用和统一 AI 助手入口。
```

## 技术栈
```text
Java、Spring Boot、Spring AI、DashScope、MyBatis、SQLite/MySQL、Embedding、RAG、Function Calling、Prompt Engineering/PostgreSQL + pgvector
```

## 项目亮点
```text
1. 设计并实现基于业务知识库的 RAG 问答链路，支持关键词检索、Embedding 语义检索和混合重排序，提升明厨亮灶监管问题回答的准确性。

2. 构建知识库管理模块，支持知识片段新增、修改、启用、禁用、reload，并通过 contentHash 实现 Embedding 向量复用，避免重复向量化调用。

3. 实现知识未命中复盘机制，当 RAG 无法命中时自动记录待补知识问题，并生成建议分类和补充动作，形成知识库持续优化闭环。

4. 实现知识变更审计能力，记录知识 CREATE、UPDATE、ENABLE、DISABLE 前后数据，支持追踪 AI 回答变化原因。

5. 实现可控版 Function Calling 工具调用框架，支持 AI 自动选择查询日志、知识库、知识变更记录等后端工具，并通过统一助手入口自动路由普通问答、RAG 问答和工具调用。
```

## 项目问答
### 1 你最近做过 AI 相关的东西吗？
```text
我最近基于 Spring Boot 和 Spring AI 做了一套面向智慧食堂 / 明厨亮灶场景的 AI 工程化 Demo。它不是简单调用大模型接口，而是完整做了 RAG 知识库、Embedding 检索、混合重排序、知识未命中复盘、知识库管理、向量持久化、知识变更审计，以及 Function Calling 工具调用。

比如用户问“留样台账没有提交，监管报告怎么写”，系统会先从业务知识库检索相关知识，再基于知识生成客观、审慎的报告表达；如果知识库没有命中，就不会编造答案，而是走兜底并记录为待补知识。后面我又做了统一 AI 助手入口，让系统自动判断问题应该走普通聊天、RAG 问答还是工具调用。
```

### 2 你这个 AI 项目具体怎么做的？
```text
第一层是模型调用层。我用 Spring AI 接入 DashScope OpenAI 兼容接口，支持普通 Chat 和 Embedding 调用，并且所有调用都会记录日志，包括业务类型、模型名称、输入、输出、耗时、是否成功、是否需要复核。

第二层是 RAG 知识库。我把明厨亮灶里的摄像头离线、画面遮挡、垃圾桶未盖、晨检台账、留样台账、消毒台账等业务规则沉淀成 knowledge_chunk 表。检索时同时走关键词检索和 Embedding 检索，再做混合重排序。

第三层是知识管理和向量持久化。我做了知识新增、修改、启用、禁用、reload，同时用 contentHash 判断知识是否变化。知识没变就复用数据库中的向量，知识变了才重新生成 Embedding，减少重复调用成本。

第四层是工具调用和统一入口。我实现了一版可控的 Function Calling，让模型先判断是否需要调用工具，然后 Java 后端根据 toolName 执行真实方法，比如查 AI 日志、查未命中问题、查知识库、查知识变更记录。最后做了 /ai/assistant/chat 统一入口，自动路由到普通聊天、RAG 问答或工具调用。

```

### 3 你这个 RAG 怎么避免胡编？
```text
我做了两层控制。第一，回答必须基于检索到的知识片段构建 Prompt，不让模型自由发挥。第二，如果知识库没有命中，就直接返回“当前知识库资料不足，无法准确回答”，并标记 needReview=true，同时记录为待补知识，而不是让模型强行回答。
```

### 4 关键词检索和 Embedding 检索有什么区别
```text
关键词检索适合精确词命中，比如“摄像头离线”“垃圾桶未盖”。
Embedding 检索适合理解语义相近的问题，比如用户说“后厨视频点位掉线了”，它能召回“摄像头离线说明”。
但 Embedding 也可能误召回相近知识，所以我做了混合检索和业务规则重排序。
```

### 5 你遇到过 RAG 误召回吗
```text
遇到过。比如“留样台账未提交”“晨检台账未提交”“消毒台账未提交”都包含“台账未提交、未提交、未上报”这些通用词，容易互相召回。后来我增加了台账子类型识别和互斥降权规则，比如用户明确问留样时，对晨检和消毒类知识降权，确保最终进入 Prompt 的上下文更准确。
```

### 6 你为什么要做向量持久化？
```text
一开始每次启动或 reload 都重新调用 Embedding 模型生成所有知识向量，这在知识少的时候没问题，但知识量大后会慢，也会增加调用成本。所以我新增了 knowledge_chunk_embedding 表，把向量以 JSON 存储，并用 contentHash 判断知识内容是否变化。知识没变就复用旧向量，只有变更的知识才重新生成向量。
```

### 7 Function Calling 你是怎么做的？
```text
我先做的是一版后端可控的 Function Calling。模型不直接操作数据库，而是先输出结构化的 toolName 和 arguments，Java 后端校验工具名和参数，然后执行对应 Service 方法。比如查询 AI 调用日志、查询知识库列表、查询知识变更记录等。
这样安全边界在后端，不会让模型随意访问系统数据。
```

### 8 为什么不直接让大模型回答所有问题？
```text
因为业务系统里很多问题需要准确性和可追溯性，比如监管报告表述、台账问题分类、知识变更记录、AI 调用日志等。单纯让模型回答容易编造。我的设计是：业务知识类问题走 RAG，系统数据类问题走工具调用，普通闲聊才走普通 Chat。
```
